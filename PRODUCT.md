# Разработка веб-приложения для аренды недвижимости с интерактивной картой и фильтрацией объектов

## 1. Описание проекта

Проект представляет собой учебное веб-приложение для поиска и аренды объектов недвижимости. Пользователи могут просматривать доступные объекты, фильтровать их по параметрам, смотреть расположение на интерактивной карте и отправлять заявки на аренду.

Администратор управляет объектами недвижимости, пользователями и заявками через простую админ-панель.

Основная цель проекта — реализовать веб-приложение с авторизацией, разделением ролей, CRUD-операциями, фильтрацией объектов, интерактивной картой и базовой бизнес-логикой обработки заявок на аренду.

---

## 2. Роли пользователей

В системе предусмотрены две роли:

- обычный пользователь;
- администратор.

### 2.1. Обычный пользователь

Обычный пользователь может:

- зарегистрироваться в системе;
- войти в аккаунт;
- просматривать список объектов недвижимости;
- использовать фильтры для поиска объектов;
- смотреть объекты на интерактивной карте;
- открывать страницу конкретного объекта;
- отправлять заявку на аренду;
- просматривать свои заявки в личном кабинете;
- отменять свои заявки, если они ещё не обработаны администратором.

### 2.2. Администратор

Администратор может:

- входить в административную панель;
- добавлять новые объекты недвижимости;
- редактировать существующие объекты;
- скрывать или удалять объекты;
- просматривать список пользователей;
- блокировать и разблокировать пользователей;
- просматривать заявки на аренду;
- изменять статус заявок;
- управлять состоянием объектов недвижимости.

---

## 3. Основные сущности системы

## 3.1. Пользователь

Пользователь — это зарегистрированный человек, который может работать с системой в зависимости от своей роли.

### Основные поля пользователя

- идентификатор;
- имя;
- email;
- пароль;
- роль;
- статус аккаунта;
- дата регистрации.

### Роли пользователя

- `user` — обычный пользователь;
- `admin` — администратор.

### Статусы аккаунта

- `active` — активный пользователь;
- `blocked` — заблокированный пользователь.

### Бизнес-логика пользователя

Email пользователя должен быть уникальным. При регистрации пользователь получает роль `user`. Роль `admin` нельзя получить через обычную регистрацию.

Пароль должен храниться в зашифрованном виде. Заблокированный пользователь может войти в систему только при необходимости просмотра истории, но не может отправлять новые заявки на аренду.

Администратор может заблокировать пользователя, если он нарушает правила сервиса. Уже созданные заявки заблокированного пользователя остаются в системе.

---

## 3.2. Объект недвижимости

Объект недвижимости — это квартира, дом, комната, студия или другой объект, который можно арендовать через систему.

### Основные поля объекта недвижимости

- идентификатор;
- название;
- описание;
- тип объекта;
- город;
- район;
- адрес;
- цена за месяц;
- количество комнат;
- площадь;
- этаж;
- общее количество этажей;
- наличие мебели;
- наличие парковки;
- возможность проживания с животными;
- координаты;
- фотографии;
- статус объекта;
- дата создания;
- дата обновления.

### Типы объектов недвижимости

- квартира;
- дом;
- комната;
- студия.

### Статусы объекта недвижимости

- `available` — объект доступен для аренды;
- `booked` — объект забронирован;
- `hidden` — объект скрыт администратором.

### Бизнес-логика объекта недвижимости

Пользователям отображаются доступные и забронированные объекты. Скрытые объекты не отображаются в пользовательской части сайта, но видны администратору.

Объект со статусом `booked` может отображаться на сайте, но пользователь не может отправить на него новую заявку.

У объекта могут быть координаты для отображения на интерактивной карте. Если координаты не указаны, объект не отображается на карте, а в форме администратора есть подсказка заполнить координаты.

Цена, площадь, количество комнат и этаж не могут быть отрицательными значениями.

---

## 3.3. Заявка на аренду

Заявка на аренду создаётся пользователем, когда он хочет арендовать конкретный объект недвижимости.

### Основные поля заявки

- идентификатор;
- пользователь;
- объект недвижимости;
- сообщение пользователя;
- желаемая дата заселения;
- статус заявки;
- дата создания;
- дата обновления.

### Статусы заявки

- `new` — новая заявка;
- `in_review` — заявка находится на рассмотрении;
- `approved` — заявка одобрена;
- `rejected` — заявка отклонена;
- `cancelled` — заявка отменена пользователем.

### Бизнес-логика заявки

Пользователь может отправить заявку только на объект со статусом `available`.

Пользователь не может отправить несколько активных заявок на один и тот же объект. Активными считаются заявки со статусами `new` и `in_review`.

После создания заявка получает статус `new`.

Пользователь может отменить свою заявку только до того, как она будет одобрена или отклонена администратором.

Администратор может изменить статус заявки. Если заявка одобрена, объект недвижимости получает статус `booked`.

После одобрения одной заявки остальные активные заявки на этот же объект автоматически отклоняются.

---

## 4. Функционал обычного пользователя

## 4.1. Регистрация

Пользователь вводит имя, email и пароль.

Система проверяет:

- заполнены ли обязательные поля;
- корректно ли указан email;
- не используется ли email другим пользователем;
- соответствует ли пароль минимальным требованиям.

После успешной регистрации пользователь создаётся в системе с ролью `user`.

---

## 4.2. Авторизация

Пользователь вводит email и пароль.

Система проверяет:

- существует ли пользователь с таким email;
- правильно ли введён пароль;
- не заблокирован ли аккаунт.

После успешной авторизации пользователь получает доступ к личному кабинету и возможности отправлять заявки на аренду.

---

## 4.3. Просмотр списка объектов недвижимости

На главной странице пользователь видит список доступных объектов недвижимости.

Для каждого объекта отображаются:

- фотография;
- название;
- цена за месяц;
- город;
- район;
- количество комнат;
- площадь;
- краткое описание;
- кнопка перехода на страницу объекта.

Объекты со статусом `hidden` не отображаются пользователям.

---

## 4.4. Фильтрация объектов

Пользователь может фильтровать объекты недвижимости по нескольким параметрам.

### Доступные фильтры

- город;
- район;
- тип недвижимости;
- минимальная цена;
- максимальная цена;
- количество комнат;
- минимальная площадь;
- наличие мебели;
- наличие парковки;
- возможность проживания с животными;
- статус доступности.

### Бизнес-логика фильтрации

Если фильтр не выбран, он не влияет на результат поиска.

Фильтры могут применяться одновременно. Например, пользователь может искать квартиру в определённом районе, с двумя комнатами, парковкой и ценой не выше указанного значения.

Если подходящие объекты не найдены, система должна показать сообщение: `По вашему запросу ничего не найдено`.

Список объектов и данные на карте должны обновляться при изменении фильтров.

---

## 4.5. Интерактивная карта

На карте отображаются объекты недвижимости, подходящие под текущие фильтры.

Каждый объект отображается на карте в виде маркера.

При клике на маркер пользователь видит краткую информацию:

- название объекта;
- цена;
- адрес;
- кнопка `Подробнее`.

### Бизнес-логика карты

На карте отображаются только объекты, которые подходят под выбранные фильтры.

Если пользователь выбирает объект из списка, карта может центрироваться на координатах этого объекта.

Если у объекта не указаны координаты, он не отображается на карте.

Для карты используется Яндекс Карты JS API.

---

## 4.6. Страница объекта недвижимости

На странице конкретного объекта пользователь видит подробную информацию об объекте.

### На странице объекта отображаются

- фотографии;
- название;
- описание;
- адрес;
- город;
- район;
- цена за месяц;
- количество комнат;
- площадь;
- этаж;
- наличие мебели;
- наличие парковки;
- возможность проживания с животными;
- карта с точкой объекта;
- кнопка отправки заявки.

### Бизнес-логика страницы объекта

Если пользователь не авторизован, вместо формы заявки отображается предложение войти в аккаунт.

Если объект уже забронирован, кнопка отправки заявки недоступна.

Если пользователь уже отправил активную заявку на этот объект, повторная отправка заявки запрещена.

---

## 4.7. Отправка заявки на аренду

Авторизованный пользователь может отправить заявку на аренду объекта.

### Форма заявки содержит

- желаемую дату заселения;
- сообщение пользователя.

### Бизнес-логика отправки заявки

Заявка создаётся со статусом `new`.

Заявку можно отправить только на доступный объект.

Если пользователь уже имеет активную заявку на этот объект, система не должна создавать повторную заявку.

Если пользователь заблокирован, он не может отправлять заявки.

После создания заявки пользователь видит её в личном кабинете, а администратор — в админ-панели.

---

## 4.8. Личный кабинет пользователя

В личном кабинете пользователь может просматривать свои данные и свои заявки.

### В личном кабинете отображаются

- имя пользователя;
- email;
- список заявок;
- статус каждой заявки;
- объект недвижимости, связанный с заявкой;
- дата отправки заявки;
- желаемая дата заселения;
- кнопка отмены заявки, если заявка ещё не обработана.

### Бизнес-логика личного кабинета

Пользователь видит только свои заявки.

Пользователь не может просматривать заявки других пользователей.

Отменённая заявка остаётся в истории, но больше не считается активной.

---

## 5. Функционал администратора

## 5.1. Админ-панель

Админ-панель — отдельный раздел приложения, доступный только пользователям с ролью `admin`.

Если обычный пользователь пытается открыть админ-панель, система должна отказать в доступе.

Админ-панель должна быть простой и содержать основные разделы:

- объекты недвижимости;
- заявки;
- пользователи.

---

## 5.2. Управление объектами недвижимости

Администратор может создавать, редактировать, скрывать и удалять объекты недвижимости.

### Форма создания и редактирования объекта содержит

- название;
- описание;
- тип объекта;
- город;
- район;
- адрес;
- цену за месяц;
- количество комнат;
- площадь;
- этаж;
- общее количество этажей;
- наличие мебели;
- наличие парковки;
- возможность проживания с животными;
- координаты;
- фотографии;
- статус объекта.

### Бизнес-логика управления объектами

Объект нельзя создать без названия, города, адреса, цены и площади. Координаты можно не указывать, но без них объект не появится на карте.

Цена и площадь должны быть положительными значениями.

Администратор может скрыть объект, если он временно недоступен.

Скрытый объект не отображается пользователям, но остаётся в базе данных.

Удаление объекта выполняется физически: запись объекта удаляется из базы данных. Забронированный объект удалить нельзя, чтобы не потерять связь с одобренной заявкой.

Если объект забронирован, администратор должен видеть связанную одобренную заявку.

---

## 5.3. Управление заявками

Администратор видит список всех заявок на аренду.

### Для каждой заявки отображаются

- пользователь;
- объект недвижимости;
- дата создания заявки;
- желаемая дата заселения;
- сообщение пользователя;
- текущий статус заявки.

### Администратор может

- перевести заявку в статус `in_review`;
- одобрить заявку;
- отклонить заявку.

### Бизнес-логика управления заявками

Одобрить можно только заявку на объект со статусом `available`.

После одобрения заявки объект получает статус `booked`.

После одобрения одной заявки остальные активные заявки на этот же объект автоматически получают статус `rejected`.

Отклонение заявки не меняет статус объекта.

Отменённую пользователем заявку нельзя повторно обработать.

---

## 5.4. Управление пользователями

Администратор может просматривать список зарегистрированных пользователей.

### Для каждого пользователя отображаются

- имя;
- email;
- роль;
- статус аккаунта;
- дата регистрации;
- количество заявок.

### Администратор может

- заблокировать пользователя;
- разблокировать пользователя;
- изменить роль пользователя.

### Бизнес-логика управления пользователями

Заблокированный пользователь не может отправлять новые заявки.

Заявки заблокированного пользователя не удаляются.

Администратор не должен иметь возможность заблокировать самого себя.

Роль `admin` можно назначить только через админ-панель или напрямую через базу данных.

---

## 6. Основные пользовательские сценарии

## 6.1. Поиск объекта недвижимости

Пользователь открывает главную страницу, выбирает нужные фильтры и получает список подходящих объектов недвижимости.

Одновременно с этим на интерактивной карте отображаются маркеры объектов, которые соответствуют выбранным фильтрам.

Пользователь выбирает подходящий объект и переходит на страницу с подробной информацией.

---

## 6.2. Отправка заявки на аренду

Пользователь авторизуется, открывает страницу объекта недвижимости и отправляет заявку на аренду.

Система проверяет, что объект доступен и пользователь ещё не отправлял активную заявку на этот объект.

После успешной отправки заявка получает статус `new`.

---

## 6.3. Обработка заявки администратором

Администратор открывает список заявок в админ-панели.

Он выбирает новую заявку и переводит её в статус `in_review`, а затем принимает решение.

Если заявка одобрена, объект получает статус `booked`.

Если заявка отклонена, объект остаётся доступным.

---

## 6.4. Добавление нового объекта недвижимости

Администратор открывает раздел объектов недвижимости и создаёт новый объект.

Он указывает адрес, цену, описание, характеристики и координаты.

После сохранения объект появляется в списке доступных объектов и на интерактивной карте.

---

## 6.5. Скрытие объекта недвижимости

Администратор скрывает объект, если он временно недоступен.

После этого объект исчезает из пользовательской части сайта, но остаётся доступным в админ-панели.

---

## 7. Минимальный набор функций для MVP

Для учебного проекта достаточно реализовать следующий функционал:

1. Регистрация пользователей.
2. Авторизация пользователей.
3. Разделение ролей на `user` и `admin`.
4. Главная страница со списком объектов недвижимости.
5. Фильтрация объектов.
6. Интерактивная карта с маркерами объектов.
7. Страница конкретного объекта недвижимости.
8. Отправка заявки на аренду.
9. Личный кабинет пользователя со списком заявок.
10. Админ-панель.
11. CRUD для объектов недвижимости.
12. Просмотр и обработка заявок администратором.
13. Просмотр пользователей администратором.
14. Блокировка и разблокировка пользователей.

---

## 8. Что можно упростить в учебном проекте

В рамках учебной курсовой работы можно не реализовывать:

- реальные платежи;
- договоры аренды;
- онлайн-чат;
- загрузку документов;
- проверку личности;
- владельцев недвижимости как отдельную роль;
- сложную модерацию;
- email-уведомления;
- интеграцию с платными картографическими сервисами;
- автоматическое определение координат по адресу.

Координаты объектов можно вводить вручную в админ-панели.

Фотографии объектов можно хранить как ссылки или загружать локально.

Данные объектов недвижимости можно заранее добавить через админ-панель или подготовить тестовые записи в базе данных.

- редактирование объекта;
- список заявок;
- страница заявки;
- список пользователей.

## 10. Итоговая логика работы системы

Обычный пользователь заходит на сайт, ищет подходящий объект недвижимости с помощью фильтров и интерактивной карты, открывает страницу объекта и отправляет заявку на аренду.

После отправки заявки пользователь может отслеживать её статус в личном кабинете.

Администратор через админ-панель добавляет и редактирует объекты недвижимости, следит за пользователями и обрабатывает заявки.

Если администратор одобряет заявку, объект становится забронированным и больше не доступен для новых заявок.

Таким образом, веб-приложение демонстрирует основные возможности современной информационной системы: регистрацию, авторизацию, разделение ролей, CRUD-операции, фильтрацию данных, работу с интерактивной картой и обработку бизнес-процессов через статусы.
