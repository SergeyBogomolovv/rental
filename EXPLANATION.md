# EXPLANATION: как устроен проект Rental Map

Этот файл нужен для подготовки к защите курсовой. Он написан простым языком: что делает проект, из каких частей состоит, зачем нужны библиотеки и как объяснить основные механики.

## 1. Что это за проект

Rental Map - это учебное веб-приложение для поиска и аренды недвижимости.

В проекте есть две стороны:

- обычный пользователь ищет объект, смотрит карту, открывает страницу объекта и отправляет заявку;
- администратор управляет объектами, пользователями и заявками.

Пример сценария:

1. Пользователь открывает сайт.
2. Видит каталог квартир, домов, комнат и студий.
3. Фильтрует объекты по городу, цене, комнатам и другим параметрам.
4. Открывает объект.
5. Если он авторизован, отправляет заявку.
6. Администратор видит заявку и может взять ее в работу, одобрить или отклонить.
7. Если заявка одобрена, объект становится забронированным.

## 2. Общая архитектура

Проект разделен на две основные части:

```text
backend/   серверная часть
frontend/  клиентская часть
```

Backend отвечает за данные и правила:

- хранит пользователей;
- хранит объекты недвижимости;
- хранит заявки;
- проверяет права доступа;
- выдает API для frontend.

Frontend отвечает за интерфейс:

- показывает страницы;
- отправляет запросы на backend;
- отображает ошибки и успешные действия;
- показывает карту;
- дает пользователю нажимать кнопки, заполнять формы и переходить между страницами.

Можно объяснить так:

> Frontend - это "витрина" и кнопки. Backend - это "касса, склад и охрана", где проверяют права, сохраняют данные и решают, можно ли выполнить действие.

## 3. Что такое API

API - это набор адресов, по которым frontend общается с backend.

Например:

- `GET /api/properties/` - получить список объектов;
- `POST /api/auth/token/` - войти в аккаунт;
- `POST /api/requests/` - создать заявку;
- `POST /api/admin/requests/{id}/approve/` - одобрить заявку.

Frontend не работает с базой данных напрямую. Он отправляет HTTP-запросы на API.

Пример:

1. Пользователь нажимает "Отправить заявку".
2. React отправляет запрос на `/api/requests/`.
3. Django проверяет пользователя, объект и правила.
4. Если все хорошо, Django сохраняет заявку в базе.
5. Frontend показывает сообщение, что заявка создана.

## 4. Backend: что используется и зачем

Backend написан на Python и Django.

### Python

Python - язык программирования, на котором написана серверная часть.

### Django

Django - фреймворк для веб-приложений. Он дает готовые инструменты:

- модели для работы с базой данных;
- систему пользователей;
- маршруты URL;
- админские и серверные механизмы;
- миграции базы данных.

В проекте Django используется как основа backend.

### Django REST Framework

Django REST Framework, или DRF, нужен для создания API.

Обычный Django часто отдает HTML-страницы. DRF удобен, когда backend должен отдавать JSON-данные для frontend.

Пример JSON-ответа:

```json
{
  "id": 1,
  "title": "Светлая студия у метро",
  "price_per_month": 68000,
  "status": "available"
}
```

### DRF Token Authentication

Это механизм авторизации по токену.

После входа пользователь получает специальную строку - токен. Frontend сохраняет токен в `localStorage` и отправляет его в следующих запросах.

Пример заголовка:

```text
Authorization: Token abc123...
```

Backend по этому токену понимает, какой пользователь делает запрос.

### django-filter

Библиотека для фильтрации объектов.

Она помогает обрабатывать параметры вроде:

```text
/api/properties/?city=Москва&rooms=2&max_price=100000
```

Backend получает эти параметры и возвращает только подходящие объекты.

### django-cors-headers

Frontend и backend запускаются на разных адресах:

- frontend: `http://127.0.0.1:5173`
- backend: `http://127.0.0.1:8000`

Для браузера это разные источники. CORS-настройки разрешают frontend обращаться к backend.

### drf-spectacular

Эта библиотека создает документацию API.

Swagger доступен по адресу:

```text
http://127.0.0.1:8000/api/docs/
```

На защите можно сказать:

> Swagger нужен, чтобы видеть список API-методов, их параметры и проверять запросы без frontend.

### SQLite

SQLite - простая база данных в одном файле `db.sqlite3`.

Для учебного проекта это удобно:

- не нужно ставить отдельный сервер базы данных;
- база хранится локально;
- легко запускать проект.

Внутри `backend/` есть несколько Django-приложений.

Статусы:

- район;

У заявки есть:

## 6. Модели, сериализаторы и views

В Django-проекте часто встречаются три важных понятия.

### Models

Модели описывают таблицы базы данных.

Например, модель `Property` означает таблицу объектов недвижимости.

Если в модели есть поле:

```python
price_per_month = models.PositiveIntegerField()
```

это значит, что у объекта есть цена за месяц, и она хранится как положительное целое число.

### Serializers

Сериализаторы переводят данные между Python-объектами и JSON.

Когда backend отправляет объект на frontend, сериализатор превращает его в JSON.

Когда frontend отправляет форму на backend, сериализатор проверяет данные и превращает JSON в объект Django.

Также в сериализаторах часто пишется валидация.

Пример валидации:

- пароль и подтверждение пароля должны совпадать;
- этаж не может быть больше общего количества этажей;
- нельзя создать заявку на занятый объект.

### Views / ViewSets

Views обрабатывают HTTP-запросы.

Например:

- получить список объектов;
- создать заявку;
- отменить заявку;
- заблокировать пользователя.

ViewSet - это удобный класс DRF, который объединяет несколько действий: список, детальная страница, создание, изменение и удаление.

## 7. Как работает регистрация

Пользователь открывает страницу регистрации и вводит:

- имя;
- email;
- пароль;
- подтверждение пароля.

Дальше происходит следующее:

1. Frontend отправляет данные на `/api/auth/register/`.
2. Backend проверяет email, пароль и совпадение паролей.
3. Если все корректно, создается пользователь.
4. Пароль не хранится обычным текстом. Django сохраняет его в зашифрованном виде, точнее как хеш.
5. Пользователь получает роль `user`.
6. Backend создает токен.
7. Frontend сохраняет токен и считает пользователя авторизованным.

Важно для защиты:

> Пользователь не может сам зарегистрироваться администратором. Роль `admin` назначается только через админ-панель или напрямую в базе.

## 8. Как работает вход

Пользователь вводит email и пароль.

1. Frontend отправляет запрос на `/api/auth/token/`.
2. Backend ищет пользователя по email.
3. Django проверяет пароль.
4. Если пароль правильный, backend возвращает токен и данные пользователя.
5. Frontend сохраняет токен в `localStorage`.
6. При следующих запросах frontend добавляет токен в заголовок `Authorization`.

Если пароль неправильный, backend возвращает ошибку.

## 9. Что такое токен и зачем он нужен

Токен - это строка, которая доказывает, что пользователь уже вошел.

Без токена backend не знает, кто отправил запрос.

Например:

- открыть каталог можно без токена;
- отправить заявку можно только с токеном;
- открыть админ-панель можно только с токеном администратора.

Токен работает примерно как временный пропуск:

> Пользователь один раз показывает email и пароль, получает пропуск-токен, а дальше показывает этот токен при каждом защищенном запросе.

## 10. Как работает выход

Когда пользователь нажимает кнопку выхода:

1. Frontend удаляет токен из `localStorage`.
2. Очищает данные пользователя.
3. Очищает кэш запросов React Query.
4. Интерфейс снова показывает кнопки "Войти" и "Регистрация".

После выхода frontend больше не отправляет токен, поэтому защищенные действия становятся недоступны.

## 11. Роли и защита доступа

В проекте есть обычные страницы и админские страницы.

Обычный пользователь может:

- смотреть каталог;
- смотреть объект;
- отправлять заявки;
- смотреть свой кабинет;
- отменять свои активные заявки.

Администратор может:

- создавать и редактировать объекты;
- скрывать и удалять объекты;
- смотреть всех пользователей;
- блокировать пользователей;
- менять роли;
- обрабатывать заявки.

Защита есть на двух уровнях.

### На frontend

Компонент `ProtectedRoute` проверяет:

- вошел ли пользователь;
- является ли он администратором.

Если пользователь не вошел, его отправляют на страницу входа.

Если обычный пользователь пытается открыть `/admin`, frontend возвращает его на главную.

### На backend

Backend тоже проверяет права.

Это важнее, чем frontend-проверка.

Почему?

Пользователь может попытаться отправить запрос напрямую, не через интерфейс. Поэтому backend обязан сам проверять, имеет ли пользователь право на действие.

Например, админские API защищены permission-классом `IsAdminRole`.

Он проверяет:

- пользователь авторизован;
- у пользователя роль администратора;
- аккаунт активен;
- аккаунт не заблокирован.

На защите можно сказать:

> Frontend скрывает лишние кнопки для удобства, но настоящая защита находится на backend.

## 12. Блокировка пользователя

Администратор может заблокировать пользователя.

Заблокированный пользователь:

- может войти;
- может смотреть свою историю;
- не может отправлять новые заявки.

Это сделано потому, что уже созданные заявки не удаляются. Пользователь может видеть историю, но не может выполнять новые действия.

Администратор не может заблокировать самого себя. Это защита от ситуации, когда последний администратор случайно лишает себя доступа.

## 13. Как работает каталог объектов

Главная страница делает запрос:

```text
GET /api/properties/
```

Backend возвращает список объектов, кроме скрытых.

Для каждого объекта показывается:

- фото;
- название;
- цена;
- город и район;
- статус;
- комнаты;
- площадь;
- кнопка "Подробнее".

Если объект `hidden`, обычный пользователь его не видит.

Если объект `booked`, он виден, но на него нельзя отправить новую заявку.

## 14. Как работает фильтрация

Фильтры находятся на странице каталога.

Пользователь может выбрать:

- город;
- район;
- тип объекта;
- минимальную и максимальную цену;
- комнаты;
- минимальную площадь;
- мебель;
- парковку;
- животных;
- статус.

Когда фильтр меняется, frontend отправляет запрос с query-параметрами.

Пример:

```text
GET /api/properties/?city=Москва&rooms=2&has_parking=true
```

Backend через `django-filter` возвращает только подходящие объекты.

Если ничего не найдено, frontend показывает:

```text
По вашему запросу ничего не найдено
```

## 15. Как работает карта

Карта сделана с помощью Яндекс Карт JS API.

Ключ API задается во frontend `.env` через `VITE_YANDEX_MAPS_API_KEY`.

На карте показываются только объекты с координатами.

Координаты - это:

- `latitude` - широта;
- `longitude` - долгота.

Если координат нет, объект остается в каталоге, но на карте не появляется.

Когда меняются фильтры, список объектов и маркеры на карте обновляются.

При клике на маркер открывается popup:

- название;
- цена;
- адрес;
- ссылка "Подробнее".

## 16. Как работает страница объекта

Страница объекта открывается по адресу:

```text
/properties/{id}
```

Например:

```text
/properties/1
```

Frontend берет `id` из адреса и запрашивает:

```text
GET /api/properties/1/
```

На странице показывается:

- фото;
- название;
- статус;
- цена;
- адрес;
- тип;
- комнаты;
- площадь;
- этаж;
- описание;
- удобства;
- карта с точкой;
- форма заявки.

Если пользователь не вошел, вместо формы показывается предложение войти.

Если объект забронирован, форма недоступна.

Если у пользователя уже есть активная заявка на этот объект, повторная заявка запрещена.

## 17. Как работает создание заявки

Пользователь заполняет:

- желаемую дату заезда;
- сообщение владельцу.

Frontend отправляет:

```text
POST /api/requests/
```

Backend проверяет:

- пользователь авторизован;
- пользователь не заблокирован;
- объект существует;
- объект свободен;
- у пользователя нет активной заявки на этот же объект.

Если все хорошо, создается заявка со статусом `new`.

Активные статусы:

- `new`;
- `in_review`.

Если заявка уже `approved`, `rejected` или `cancelled`, она не считается активной.

## 18. Как работает личный кабинет

Кабинет доступен только авторизованному пользователю.

В кабинете отображается:

- имя;
- email;
- список заявок;
- статус заявки;
- объект;
- сообщение;
- дата создания;
- желаемая дата заезда;
- кнопка отмены для активной заявки.

Пользователь видит только свои заявки.

Это важно:

> Даже если пользователь попробует вручную запросить чужую заявку, backend фильтрует заявки по текущему пользователю.

## 19. Как работает отмена заявки

Пользователь может отменить только активную заявку.

Когда он нажимает "Отменить", frontend отправляет:

```text
POST /api/requests/{id}/cancel/
```

Backend проверяет статус.

Если статус `new` или `in_review`, заявка становится `cancelled`.

Если заявка уже `approved` или `rejected`, отменить ее нельзя.

## 20. Как работает админ-панель

Админ-панель доступна по адресу:

```text
/admin
```

Она видна только пользователям с ролью `admin`.

Основные разделы:

- обзор;
- объекты;
- заявки;
- пользователи.

## 21. Управление объектами в админке

Администратор может:

- создать объект;
- редактировать объект;
- скрыть объект;
- показать скрытый объект;
- удалить объект.

При создании и редактировании используются поля:

- название;
- описание;
- тип;
- город;
- район;
- адрес;
- цена;
- комнаты;
- площадь;
- этаж;
- общее количество этажей;
- мебель;
- парковка;
- животные;
- координаты;
- URL фото;
- статус.

Если объект скрыть, он исчезает из публичного каталога, но остается в админке.

Если объект удалить, запись физически удаляется из базы.

Забронированный объект удалить нельзя, потому что с ним связана одобренная заявка.

У забронированного объекта администратор видит информацию о связанной одобренной заявке.

## 22. Управление заявками в админке

Администратор видит все заявки всех пользователей.

Для каждой заявки показывается:

- статус;
- объект;
- email пользователя;
- дата создания;
- желаемая дата заезда;
- сообщение;
- доступные действия.

Администратор может:

- перевести новую заявку в `in_review`;
- одобрить заявку;
- отклонить заявку.

Если заявка одобрена:

1. Заявка получает статус `approved`.
2. Объект получает статус `booked`.
3. Другие активные заявки на этот же объект автоматически получают статус `rejected`.

Если заявка отклонена, объект остается свободным.

Отмененную или уже обработанную заявку нельзя обработать повторно.

## 23. Управление пользователями

В разделе пользователей администратор видит:

- email;
- роль;
- статус аккаунта;
- дату регистрации;
- количество заявок.

Администратор может:

- заблокировать пользователя;
- разблокировать пользователя;
- сделать пользователя администратором;
- сделать администратора обычным пользователем.

Но текущий администратор не может:

- заблокировать самого себя;
- снять с себя роль администратора.

Это защита от случайной потери доступа к админке.

## 24. Frontend: что используется и зачем

Frontend написан на React.

### React

React - библиотека для создания интерфейса.

В React интерфейс собирается из компонентов.

Примеры компонентов:

- `PropertyCard` - карточка объекта;
- `PropertyMap` - карта;
- `ProtectedRoute` - защита страниц;
- `AppLayout` - общий layout с верхним меню.

### Vite

Vite - инструмент для запуска frontend в режиме разработки и сборки production-версии.

Команды:

```bash
npm run dev
npm run build
```

`npm run dev` запускает сайт локально.

`npm run build` проверяет, что проект собирается.

### React Router

React Router отвечает за страницы и переходы.

Примеры маршрутов:

- `/` - каталог;
- `/properties/:id` - страница объекта;
- `/login` - вход;
- `/register` - регистрация;
- `/cabinet` - кабинет;
- `/admin` - админка.

### TanStack Query

TanStack Query помогает работать с серверными данными.

Он умеет:

- загружать данные;
- хранить кэш;
- обновлять данные после изменений;
- показывать состояние загрузки.

Например, после создания заявки проект обновляет список заявок через `invalidateQueries`.

### Axios

В проекте есть общий API-клиент:

frontend/src/api/client.js

Используется для карты.

Яндекс Карты рисуют карту, маркеры и всплывающие карточки объектов.

## 25. Как frontend хранит пользователя

После входа backend возвращает:

- токен;
- данные пользователя.

Frontend сохраняет токен в `localStorage` под ключом:

```text
rental_token
```

Данные пользователя хранятся в React-состоянии внутри `AuthContext`.

`AuthContext` нужен, чтобы любая часть приложения могла узнать:

- пользователь вошел или нет;
- кто текущий пользователь;
- является ли он администратором;
- идет ли загрузка;
- как выполнить вход;
- как выполнить выход.

## 26. Что происходит при запросе с frontend

Пример: пользователь открыл кабинет.

1. Компонент `CabinetPage` делает запрос на `/requests/`.
2. Axios добавляет токен в заголовок.
3. Backend по токену определяет пользователя.
4. Backend возвращает только заявки этого пользователя.
5. React отображает список.

Если токена нет, защищенная страница отправляет пользователя на `/login`.

## 27. Важные правила бизнес-логики

Эти правила можно перечислить на защите:

- скрытые объекты не видны обычным пользователям;
- забронированные объекты видны, но заявка на них недоступна;
- нельзя создать заявку на занятый объект;
- нельзя создать две активные заявки на один и тот же объект;
- заблокированный пользователь не может создать заявку;
- пользователь видит только свои заявки;
- администратор видит все заявки;
- при одобрении заявки объект становится забронированным;
- остальные активные заявки на этот объект автоматически отклоняются;
- администратор не может заблокировать сам себя;
- забронированный объект нельзя удалить.

## 32. Возможные вопросы на защите

### Почему выбран Django?

Django дает готовую структуру для backend: модели, пользователей, миграции, работу с базой и безопасность. Для учебного проекта это быстрее и надежнее, чем писать все с нуля.

### Почему frontend отдельно от backend?

Такой подход ближе к современным веб-приложениям. Backend отдает данные через API, а frontend отдельно отвечает за интерфейс.

### Где хранится пароль?

Пароль не хранится в открытом виде. Django сохраняет хеш пароля. При входе Django сравнивает введенный пароль с хешем.

### Что такое токен?

Токен - это строка, по которой backend узнает авторизованного пользователя. Frontend отправляет токен в заголовке `Authorization`.

### Почему нельзя доверять только frontend?

Потому что пользователь может отправить запрос напрямую, минуя интерфейс. Поэтому все важные проверки есть на backend.

### Как запрещен доступ в админку обычному пользователю?

На frontend обычный пользователь не видит пункт "Админка" и не проходит через `ProtectedRoute`. На backend админские API проверяют роль через permission-класс.

### Что происходит при одобрении заявки?

Заявка становится `approved`, объект становится `booked`, а остальные активные заявки на этот объект становятся `rejected`.

### Почему есть статус `hidden`?

Он нужен, чтобы администратор мог временно убрать объект из публичного каталога, не удаляя его из базы.

### Почему используется SQLite?

SQLite простая и не требует отдельной установки сервера базы данных. Для учебного проекта этого достаточно.

### Что делает React Query?

Он загружает данные с backend, кэширует их и помогает обновлять интерфейс после действий вроде создания заявки или изменения объекта.

### Что делает Axios?

Axios отправляет HTTP-запросы на backend. В проекте он также автоматически добавляет токен авторизации.

### Что такое serializer?

Serializer переводит данные между Django-моделью и JSON, а также проверяет входящие данные.

### Что такое migration?

Migration - это инструкция для базы данных, как создать или изменить таблицы.

## 33. Короткий рассказ о проекте на защите

Можно сказать так:

> Мой проект - веб-приложение для аренды недвижимости. Оно состоит из backend на Django REST Framework и frontend на React. Пользователь может искать объекты, фильтровать их, смотреть карту и отправлять заявки. Администратор управляет объектами, пользователями и заявками. Авторизация реализована через DRF Token Authentication: после входа пользователь получает токен, который frontend отправляет в защищенных запросах. На backend реализованы роли, статусы аккаунтов, фильтрация, бизнес-логика заявок и проверки прав доступа. Frontend использует React Router для страниц, TanStack Query для загрузки данных, Axios для API-запросов и Яндекс Карты JS API для карты.

## 34. Что важно помнить

Самые важные идеи:

- frontend показывает интерфейс;
- backend хранит данные и проверяет правила;
- API связывает frontend и backend;
- токен подтверждает авторизацию;
- роли разделяют возможности пользователя и администратора;
- бизнес-логика заявок защищает систему от неправильных действий;
- скрытые объекты не удаляются, а просто не показываются пользователям;
- одобрение заявки меняет статус объекта на забронированный.
